Recommendation · Buy now — before Search Console and before Issue 01 sends

The argument is sequencing, not vanity.

Two launches sit send-ready, each one Vini-action from firing: Search Console registration (the 06-07 sitemap packet, never submitted) and the Substack Issue 01 send. Both bind permanently to whatever URL exists the moment they fire — Search Console binds a property, Substack binds a from-address and every archive link it ever creates. Fire them on education3881.github.io/madar and the migration is paid twice: once in redirects, once in re-registration and reader confusion. Today the site is 18 days old, has 27 URLs, and effectively zero inbound links — migration cost is at its lifetime minimum this week and steps up at the first binding. And the URL is brand chrome: trust is the moat, and education3881 reads as a throwaway account to exactly the readers we most want — ministries, researchers, people who notice URLs.

SEO / discoverability — what moves, what breaks
Near-zero loss today: no Search Console property exists, the sitemap was never submitted, inbound ≈ 0. There is no rank to protect — only future rank to site correctly. GitHub 301-redirects the github.io origin to the custom domain once configured (docs-confirmed at the domain level; deep-path behaviour on project pages gets a curl -I spot-check post-cutover before we declare done). All 27 canonicals and 12 EN↔AR hreflang pairs regenerate from one config change — verified today: no hardcoded base path anywhere in src; the origin lives in exactly three places (astro.config, the static sitemap, one robots.txt line). Clean-cut bonus: register Search Console on the new domain only — no property migration, no dual-property limbo.
The cutover (Web Developer's sequence — ~90 minutes of work + DNS/cert wait)
(1) Verify the domain in GitHub before any DNS exists — prevents takeover squatting. (2) DNS: apex A records to GitHub's four IPs + www CNAME, both, so GitHub auto-redirects between them. (3) CNAME file into web/public/ — the Actions deploy must carry it, settings-only config gets overwritten. (4) astro.config: site → new domain, base → '/' — every internal URL changes; full rebuild + page-by-page QA. (5) Fold the standing @astrojs/sitemap migration into the same move — one URL-set change, never two. (6) HTTPS: wait for the cert, then the Enforce-HTTPS toggle — the site serves plain HTTP until that second step. (7) Post-cutover: deep-path redirect checks, then Search Console + sitemap submission on the new domain.
What else "professional" buys
